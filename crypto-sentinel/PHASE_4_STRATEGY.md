# 🧠 Phase 4 (完全自律化) への戦略的考察と最終確認

これまでの資料に基づき、システムが「人間（ユーザー）のサポートツール」から「資産を運用する自律エージェント」へと飛躍する Phase 4 における技術的課題とロードマップを考察します。

## 1. 構造的パラダイムシフト（「誰が」システムを動かすか）

**現状 (Phase 3まで)**:
- ユーザーがダッシュボードを開き、「更新」ボタンを押すことで `/api/analyze` が実行され、Supabase に保存される。主導権は「人間」にある。

**Phase 4 のあるべき姿**:
- **Cron トリガー（定期実行）**: Cloudflare Workers の `Cron Triggers`、または Supabase の `pg_cron` / `Edge Functions` を用いて、ブラウザが開かれていなくてもシステムが 1時間おきに自動で目覚める必要がある。
- ダッシュボード（UI）の役割は「実行の起点」から、「AIエージェントの活動報告を見るモニタリングルーム」へと変化する。

## 2. AI (LLM) の役割と配置の再定義

現在 `lib/analysis.ts` にハードコードされている「RSIが30以下なら+40点」といった静的なロジックをどう扱うかが鍵となります。

- **役割分担の推奨**:
  - **計算層**: 現状の `lib/analysis.ts` はそのまま「市場のファクト（事実）」を抽出する計算層として残す（ハルシネーションを防ぐため）。
  - **思考層 (LLM)**: Phase 3.5 で構築予定の `pgvector` による過去の類似データと、`analysis.ts` が弾き出した「Score」を LLM に投げ、「本当にエントリーすべきか？」の最終判断（承認・却下）を委ねる。
  - プロンプト例: *"現在のオシレーターは買いを示していますが、過去の類似相場では直後に急落しています。全体情勢を加味した上で、エントリーの可否をJSONで出力してください。"*

## 3. リスク管理と セキュリティ（最重要事項）

AI に資金を預けるにあたって、現在の構成に以下のアップデートが必須です。

### A. ヒューマン・イン・ザ・ループ (HITL) の導入
最初から完全自動化（APIで勝手に買う）することは資金喪失のリスクが高すぎます。
1. AIが「買い」と判断した場合、`trades` テーブルへ `status: 'PENDING_APPROVAL'` として保存する。
2. 同時に Resend を使ってユーザーへ承認要求メール（またはダッシュボードへの通知）を送信。
3. ユーザーがダッシュボード上で「Approve（承認）」ボタンを押して初めて、実際の Binance API へ注文リクエストが飛ぶ仕組みにする。

### B. 安全装置（キルスイッチ＆サーキットブレーカー）
1. **ストップロス制限**: Binance API で発注する際、必ず同時に逆指値（ストップロス）注文を入れるようにプログラムを組む。
2. **連続取引制限**: 「1日最大1回まで」「既存の建玉がある場合は新規エントリー不可」など、DBレベル（Row Level Security 制約や Edge Function 内）で安全ストッパーを設ける。

### C. 実行権限の分離
- API を叩く Cloudflare Worker / Supabase Edge Functions は、Binance の `API_KEY` と `SECRET_KEY` を厳重に持つ必要があります。これは絶対にフロントエンド（`NEXT_PUBLIC_...`）には露出させず、サーバーサイドのみで管理してください。

## 4. Phase 4 に向けた最短ロードマップ

1. **(設計)** Cloudflare Workers に `wrangler.toml` で `[triggers] crons = ["0 * * * *"]` (1時間に1回) を設定し、自動でデータ収集・分析を回るようにする。
2. **(AI連携)** 分析完了時スコアが閾値を超えた場合などに、バックエンドから LLM API（OpenAI 等）またはローカルの Ollama に判断を仰ぐ処理を追加。
3. **(承認フロー)** LLM がGOサインを出したら、DB に `PENDING` トレードを作り、メール通知。
4. **(実行)** ユーザー承認後、Binance API 経由で注文実行。

## 💡 考察の総括
現在の**「確実にデータをDBに永続化できる基盤」**と**「複合テクニカル指標の計算ロジック」**はすでに商用レベルまで仕上がっています。ここからは「ゼロから作る」のではなく、「既存の盤石なデータを、いかに安全に AI 에料理させるか」という設計力が問われる、非常にエキサイティングなステージとなります。
